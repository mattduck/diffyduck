* Log Mode Performance Optimization

** Problem Statement

When viewing many commits in log mode (e.g., 1000 commits), several operations become slow:
- Pressing shift-tab to expand all commits/files blocks for a long time
- Pressing G to jump to end shows visibly slow line rendering
- General sluggishness when navigating expanded views

** Root Cause Analysis

The bottleneck is in =buildRows()= (view.go:251), which does multiple full passes over
all commits and files:

*** Pass 1: Calculate max header/stats widths (lines 264-290)
#+begin_src go
for commitIdx, commit := range m.commits {
    for fileIdx := startIdx; fileIdx < endIdx; fileIdx++ {
        fp := m.files[fileIdx]
        header := formatFileHeader(fp)      // string formatting
        w := displayWidth(header)           // unicode width calc
        added, removed := countFileStats(fp) // count diff lines
        // ... track max widths
    }
}
#+end_src

*** Pass 2: Calculate structural diff widths (lines 310-315)
#+begin_src go
for fileIdx := range m.files {
    w := m.structuralDiffMaxContentWidth(fileIdx)
    // ... track max width
}
#+end_src

*** Pass 3: Calculate commit header column widths (lines 335-376)
#+begin_src go
for commitIdx := range m.commits {
    for i := startIdx; i < endIdx; i++ {
        added, removed := countFileStats(m.files[i])  // DUPLICATE work
        // ... track max widths
    }
    formatShortRelativeDate(...)  // date formatting
}
#+end_src

*** Pass 4: Actually build the rows (lines 379+)
Finally iterates again to create displayRow structs for all visible content.

** Impact

With 1000 commits and ~5 files per commit = 5000 files:
- Pass 1: 5000 iterations with formatFileHeader + displayWidth + countFileStats
- Pass 2: 5000 iterations with structuralDiffMaxContentWidth
- Pass 3: 1000 + 5000 iterations with countFileStats (called AGAIN)
- Pass 4: Creates thousands of displayRow structs

=countFileStats()= is called twice per file across different passes.

** Potential Solutions

*** Option 1: Cache computed values
Pre-compute and cache expensive calculations:
- File header strings and their display widths
- File stats (added/removed counts)
- Structural diff widths

Store in FilePair or a separate cache map. Invalidate when file content changes.

*** Option 2: Lazy/estimated width calculation
- Only calculate exact widths for visible files
- Use conservative estimates for alignment (e.g., assume max 999 lines)
- Accept slightly imperfect alignment for off-screen content

*** Option 3: Incremental row updates
Instead of rebuilding all rows when fold levels change:
- Track which commits/files changed
- Only rebuild affected rows
- Splice into existing row cache

*** Option 4: Virtualized row building
Don't build all rows upfront:
- Only build rows for current viewport + buffer
- Build more rows on scroll
- Would require significant architecture changes

** Recommended Approach

Start with Option 1 (caching) as it's lowest risk:

1. Add =cachedStats= field to Model or FilePair:
   #+begin_src go
   type fileCachedStats struct {
       header       string
       headerWidth  int
       added        int
       removed      int
       structWidth  int
   }
   #+end_src

2. Compute once when file is added/loaded, reuse in buildRows()

3. Invalidate cache when file content changes (rare after initial load)
