** Phase 2: Inline/Word Diff [DONE]

Highlight the specific changed characters within modified lines, making it
easy to spot exactly what changed.

*** Visual Example

#+begin_example
Before (current):
─  15 fmt.Println("hello world")     │
                                      │+  15 fmt.Println("hello there")

After (with inline diff):
─  15 fmt.Println("hello [world]")   │
                                      │+  15 fmt.Println("hello [there]")
                                              highlighted ─────┘
#+end_example

The brackets represent highlighting (actual rendering uses color/style).
- Removed text: red background or strikethrough
- Added text: green background or bold

*** Algorithm Options

1. *Character-level Myers diff*
   - Most accurate, shows exact character changes
   - Can be noisy for heavily modified lines
   - Standard algorithm, well-understood

2. *Word-level diff*
   - Split on whitespace/punctuation, diff words
   - Cleaner output for prose/code
   - May miss character-level changes within words

3. *Hybrid approach*
   - Word-level first, then character-level within changed words
   - Best of both worlds
   - More complex implementation

*** Implementation

// pkg/inlinediff/diff.go

type Span struct {
    Start, End int
    Type       SpanType  // Unchanged, Added, Removed
}

// Diff compares two strings and returns spans for highlighting
func Diff(old, new string) (oldSpans, newSpans []Span)
#+end_src

*** Integration Points

- Modify =pkg/sidebyside= to pair up old/new lines for diffing
- Extend view rendering to apply inline highlights on top of line highlights
- Consider interaction with syntax highlighting (inline diff + syntax colors)

*** Performance Considerations

- *Compute once*: Calculate inline diffs when building DisplayLines, cache result
- *Skip long lines*: For very long lines (>500 chars?), skip or truncate
- *Skip dissimilar lines*: If lines are <30% similar, don't bother (whole line changed)

*** Dependencies

- None! Can work with current diff lines, no full file content needed
- Independent of syntax highlighting (but should compose with it later)

** Phase 3: Folding [DONE]

A major feature with three distinct view modes for each file:

*** Three Fold Levels

1. *Collapsed (stats only)*: Just the file header with +/- line counts
   #+begin_example
   ═══ ○ ~ internal/tui/view.go      | +142 -16 +++++++++++++++++++++--
   ═══ ○ + internal/tui/new_file.go  | +50      +++++++++++++++++++++++
   ═══ ○ - internal/tui/deleted.go   | -30      --------------
   ═══ ○ > old.go → new.go           | +5  -2   +++++--
   #+end_example

   Status indicators: =~= modified (blue), =+= added (green), =-= deleted (red), =>= renamed (blue)

2. *Normal (default)*: Standard diff view with hunk edits (what we have now)
   #+begin_example
   ▾ internal/tui/view.go  +45 -12
      10 context line                    │   10 context line
   ─  11 removed line                    │
                                         │+  11 added line
      12 context line                    │   12 context line
   #+end_example

3. *Expanded (full file)*: Complete file contents with changes highlighted inline
   - Shows all lines from the file, not just diff context
   - Changed sections are still highlighted
   - Useful for understanding surrounding context

*** Keybindings

- =TAB= - cycle fold level for current file (collapsed → normal → expanded → collapsed)
- =Shift+TAB= - cycle fold level for ALL files simultaneously
- Or consider: =TAB= folds, =Shift+TAB= unfolds (simpler mental model)

Alternative vim-style bindings (can coexist):
- =zc= - collapse current file to stats only
- =zo= - expand to normal diff view
- =zO= - expand to full file view
- =zM= - collapse all files
- =zR= - expand all files to normal

*** Dependencies

This feature requires fetching full file contents (old and new versions):

1. *Git layer for file retrieval*:
   - Get old version: =git show <parent>:<path>= or =git show <ref>^:<path>=
   - Get new version: =git show <ref>:<path>= or working tree for unstaged
   - For =git diff= (unstaged): old is index, new is working tree
   - For =git diff --cached=: old is HEAD, new is index

2. *Content storage*:
   - Store full file contents lazily (only when expanded)
   - Cache to avoid re-fetching on fold/unfold cycles
   - Memory consideration: large files in large diffs

3. *Line mapping*:
   - Map diff line numbers to full file line numbers
   - Track which lines are adds/removes/context/unchanged

*** State Tracking

#+begin_src go
type FoldLevel int

const (
    FoldCollapsed FoldLevel = iota  // stats only
    FoldNormal                      // diff hunks (default)

// Per-file state

*** Performance Considerations

- *Lazy loading*: Don't fetch full files until user expands
- *Incremental rendering*: Only render visible viewport
- *Cache aggressively*: Once fetched, keep file contents for session
- *Avoid re-parsing*: When toggling fold levels, reuse parsed content

Complexity: High (architectural, git integration, state management)
Files: internal/tui/model.go, pkg/sidebyside/types.go, pkg/git/git.go (new methods)
New consideration: May want pkg/content or similar for file content management

*** Search (less-style) [DONE]

A major feature: less-pager-style incremental search.

**** Core Behavior

- =/= - open search prompt, search forward as you type
- =?= - open search prompt, search backward
- =n= - jump to next match
- =N= - jump to previous match
- =Enter= or =Esc= - close search prompt, stay at current match

**** Key Design Decision: Search Visible Content Only

Search operates on *visible content only*, respecting fold state:
- Collapsed files: not searched (only header visible)
- Normal diff view: search the diff lines shown
- Expanded view: search the full file contents

This is like =less= - you search what you can see. If something is folded
away, it's hidden from search. User can unfold-all then search if needed.

Benefits:
- Simpler implementation (search the rendered/renderable content)
- Predictable behavior (matches = what you can see)
- Faster (no searching hidden content)

**** Visual Feedback

- Highlight all matches in visible area (dim highlight)
- Current match gets distinct highlight (reverse video or bright)
- Search should work across both sides in side-by-side view
- Show match count in status bar: "Match 3/17" or "No matches"

**** Dependencies

- *Status bar*: Required for showing search prompt and match count
  - Need input area at bottom for typing search term
  - Need status display for "Match N/M" or "Pattern not found"
  - Need to show current search term while active

**** Implementation Notes

- Consider regex support (like less) vs literal-only (simpler to start)
- Case sensitivity: =i= flag or smart-case (literal if lowercase, sensitive if mixed)
- Store search history for up-arrow recall
- Wrap around at end of file (with indicator)

**** Performance Considerations

- *Incremental search*: Update matches as user types, but debounce
- *Don't recompute on scroll*: Cache match positions, just update "current"
- *Visible-only matching*: Compute matches for viewport + buffer, not entire content
- *Avoid blocking*: Large files shouldn't freeze UI during search

Complexity: Medium-High (needs status bar first, search state, highlighting integration)
Files: internal/tui/model.go (search state), internal/tui/update.go (input handling),
       internal/tui/view.go (match highlighting), internal/tui/keys.go

** Performance Philosophy

A key concern for folding, search, and syntax highlighting. The goal is
to keep the UI responsive on every keystroke, even with large diffs.

*** Principles

1. *Lazy loading*: Don't fetch/parse until needed
   - Full file contents: only when user expands to full-file view
   - Syntax highlighting: only for visible viewport + small buffer
   - Search: compute incrementally, not all at once

2. *Render only visible*: Never compute the entire buffer
   - Viewport + small buffer for smooth scrolling
   - Folded content = zero rendering cost
   - Truncate/virtualize massive files if needed

3. *Cache aggressively*: Don't recompute on navigation
   - Parse results for syntax highlighting
   - Search match positions
   - Full file contents once fetched

4. *No blocking on keystrokes*: UI should never freeze
   - Debounce expensive operations (search as you type)
   - Consider goroutines for heavy work (file fetches, parsing)
   - Show loading state if operation takes time

5. *Profile before optimizing*: Measure actual bottlenecks
   - =go test -bench= for hot paths
   - Real-world test: large diffs (1000+ files, huge files)

*** Architecture Implications

- View rendering must be viewport-aware (already is for vertical)
- May need line virtualization for horizontal scrolling in huge files
- Consider streaming/incremental parsing for treesitter on large files
- Search should compute "next N matches" not "all matches everywhere"

This makes it difficult to write golden tests that verify styling/colors.

- =termenv.WithUnsafe()= forces ANSI output, but timing is tricky
- Package var initialization order matters: need to set renderer before styles are created
- Creating a custom renderer with =termenv.NewOutput(os.Stdout, termenv.WithUnsafe())=
  and =lipgloss.SetDefaultRenderer()= should work, but getting it to run before
  package-level style vars are initialized is problematic in test context

Potential solutions to investigate:
1. Make styles lazy (functions that return styles instead of package vars)
2. Use =TestMain= to set up the renderer before any tests run
3. Avoid testing ANSI output directly; test that the right styling functions are called
4. Use a mock renderer for tests that captures style applications

** Cursor & Gutter Styling [DONE]

*** Features Implemented

- *Cursor arrow indicator*: All row types show =➤= (fg=15, no bg) when cursor is on them

| Row Type       | Cursor State | Indicator | Gutter Content      |
|----------------|--------------|-----------|---------------------|

| Diff line      | normal       | +/-/space | line num (colored)  |

** Summary Row [DONE]

Git-style summary line at the bottom of the diff view showing aggregate stats.

*** Features Implemented

- Appears at the end of all files in all view modes (folded, normal, expanded)
- Format: =  ════ ●   N files changed, N insertions(+), N deletions(-)=
- Gutter area (════) matches dynamic =lineNumWidth()= for alignment
- Uses expanded bullet icon (●) since there's no additional content to show
- Non-bold text to visually distinguish from file headers
- Properly handles singular/plural ("1 file" vs "2 files", etc.)
- Omits zero-count sections (no "0 insertions" shown)
- Selectable with cursor like any other row
- Tab/fold toggle does nothing when cursor is on summary row (no file association)
- Cursor row shows ➤ arrow and highlighted gutter (see Cursor & Gutter Styling)

*** Implementation

Key functions in =internal/tui/view.go=:

- =formatSummaryStats(files, added, removed int) string=: Produces git-style text
- =renderSummary(...)=: Renders the summary row with proper styling
- =buildRows()=: Adds summary row at end with =isSummary: true= and =fileIndex: -1=

Changes to =internal/tui/model.go=:
- =fileAtLine()=: Returns last file info when on summary row

Changes to =internal/tui/update.go=:
- =currentFileIndex()=: Returns -1 when cursor is on summary row

*** Future Improvements

- *Performance*: =currentFileIndex()= calls =buildRows()= which is expensive; could cache rows or add a simpler check
- *Binary files*: Currently counts binary files the same as text; could show "N binary files" separately
- *Mode changes*: Git shows mode changes (permissions) in its summary; we don't track these
- *Summary row cursor highlight*: The =═══= prefix gets highlighted but the bullet doesn't; could highlight the whole prefix area consistently
- *Alignment*: Summary row doesn't align with file headers' stats column; could pad to match =maxHeaderWidth= for visual consistency in folded view
