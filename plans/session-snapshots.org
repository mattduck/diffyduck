#+TITLE: Session Snapshots (Virtual Commits)
#+DATE: 2026-02-03

* Overview

Add the ability to snapshot the working tree state during a diff session, showing
incremental changes as virtual commits. This enables tracking progress while
working - like reviewing PR commits one-by-one, but without requiring actual commits.

* Mental Model

- Program starts in diff mode, automatic snapshot S0 taken in background
- User works on their code...
- Press key: see diff S0→current as a virtual commit at top, snapshot S1 taken
- User works more...
- Press key: see diff S1→current as another virtual commit at top, snapshot S2 taken
- And so on...

Each keypress shows "what changed since I last looked" and advances the baseline.

* Applicability

This feature only makes sense when diffing working tree against a commit ref.
We need a commit as the base for the snapshot parent chain.

| Mode               | Applicable? | Notes                                    |
|--------------------+-------------+------------------------------------------|
| ModeDiffUnstaged   | No*         | Base is index (not a commit)             |
| ModeDiffCached     | No*         | Diffing index, not working tree          |
| ModeDiffRefs (1)   | Yes         | ref1 vs working tree - ref1 is base      |
| ModeDiffRefs (2)   | No          | Two arbitrary refs, neither is "current" |
| ModeShow           | No          | Historical commit                        |
| Log mode           | No          | Browsing history                         |
| Pager mode         | No          | No git access                            |

*Note: ModeDiffUnstaged and ModeDiffCached were originally marked Yes, but
with the parent chain approach, we need a commit as the base. See "Open
Questions" in the Persistent Snapshots section for discussion.

The Model should store a flag like `snapshotsEnabled bool` based on this,
plus `baseRef string` for the snapshot chain parent.

* Snapshot Mechanism

** Creating Snapshots Without Affecting Git State

Use a temporary index file to create dangling commits:

#+BEGIN_SRC bash
# Create temporary index
export GIT_INDEX_FILE=/tmp/dfd-snapshot-$$-$timestamp
git add -A  # or -u if not including untracked
tree=$(git write-tree)
commit=$(git commit-tree $tree -m "dfd snapshot")
rm $GIT_INDEX_FILE
echo $commit  # store this SHA
#+END_SRC

This approach:
- Does NOT touch the real index
- Does NOT create refs (no branches, tags, stash entries)
- Creates a dangling commit (invisible to normal git operations)
- Gets garbage collected automatically (default: 2 weeks for unreachable objects)
- Works even while other processes modify files

** Diffing Between Snapshots

To diff between two snapshots:
#+BEGIN_SRC bash
git diff $snapshot1_sha $snapshot2_sha
#+END_SRC

To diff a snapshot against current working tree:
#+BEGIN_SRC bash
git diff $snapshot_sha
#+END_SRC

Both produce unified diff output our parser already handles.

** Handling Untracked Files

The initial command dictates whether untracked files are included:
- If `--all` / `-a` flag was used: `git add -A` (include untracked)
- Otherwise: `git add -u` (only tracked files)

Store this as `allMode bool` in the snapshot state.

* UI Design

** Display as Virtual Commits

Snapshots appear as commit nodes at the top of the view, similar to log mode:
- Use the existing CommitSet structure
- CommitInfo has pseudo-metadata:
  - SHA: abbreviated snapshot commit SHA
  - Subject: "Diff 1", "Diff 2", etc.
  - Date: actual timestamp when snapshot was taken
  - Author: could be empty or "(dfd session)"
- Files: the actual diff between previous snapshot and this one
- No details node needed (unlike real commits)

When snapshots exist, the view becomes:
#+BEGIN_EXAMPLE
┌─ Diff 2 (just now) ─────────────────────────────
│  src/file.go  +5 -2
│  ...
├─ Diff 1 (3 minutes ago) ────────────────────────
│  src/other.go  +12 -0
│  ...
├─ (original diff) ───────────────────────────────
│  <existing diff content>
└─────────────────────────────────────────────────
#+END_EXAMPLE

** Keybinding

`R` (capital R) - triggers snapshot and shows incremental diff

** Folding Behavior

- Snapshot commits can be folded/unfolded like regular commits in log mode
- Initial state: probably Normal (showing files) since user just asked to see it
- Original diff content remains below, unchanged

* Edge Cases

** No Changes Since Last Snapshot

When user presses the key but nothing changed:
- Show a status message: "No changes since last snapshot"
- Don't create a new snapshot or empty commit node

** Initial Snapshot Fails

If we can't create the initial snapshot (e.g., git errors):
- Disable the feature gracefully
- Maybe show a status message explaining why

** Large Working Tree

Creating a snapshot of a huge repo could be slow:
- Initial snapshot happens in background (Cmd)
- Show loading indicator if user presses key before initial snapshot completes?

** Multiple dfd Instances

Each instance maintains its own snapshot list in memory:
- Snapshot SHAs are just stored in Model.snapshots
- No shared state needed
- Dangling commits from different instances don't conflict

* Decisions

- Keybinding: `R` (capital R)
- Labeling: "Diff 1", "Diff 2", etc.
- No cumulative view (S0→current) for now - only incremental (Sn-1→current)
- Let git gc handle dangling commit cleanup (no explicit cleanup on exit)

* Persistent Snapshots (Cross-Session)

** Overview

Allow snapshot history to persist across dfd restarts, so you can continue
tracking incremental changes from where you left off.

** Requirements

- Opt-in via flag (e.g., ~--continue~ or ~-c~)
- When enabled, shows diff of current working state vs last persisted snapshot
- Full snapshot history persists (not just the last one)
- "Diff N" numbering continues from previous session
- Reset mechanism to clear persisted state

** Storage: Git Refs

Store snapshots as refs in the project's git repo:

#+BEGIN_EXAMPLE
refs/dfd/snapshots/0 -> sha1
refs/dfd/snapshots/1 -> sha2
refs/dfd/snapshots/2 -> sha3
#+END_EXAMPLE

Benefits:
- Discoverable via ~git for-each-ref refs/dfd/~
- Protected from garbage collection (refs keep commits alive)
- No external state file needed - everything in git
- Local only - not pushed to remotes (refs/dfd/* not in default refspec)
- Invisible to normal git commands (~git branch~, ~git tag~)

** CLI Design

#+BEGIN_SRC bash
# Start fresh (current behavior)
dfd diff

# Continue from last session's snapshot
dfd diff --continue
dfd diff -c

# Clean up persisted snapshots
dfd clean
#+END_SRC

** Behavior

*** With ~--continue~ flag:
1. Look for existing refs in ~refs/dfd/snapshots/~
2. If found:
   - Load snapshot history from refs
   - Initial view shows diff from last snapshot to current working state
   - Continue numbering from where we left off ("Diff N+1")
3. If not found:
   - Behave like normal startup (fresh session)

*** Without flag:
- Current behavior (fresh session, no persistence)
- Snapshots are still dangling commits (will be GC'd eventually)

*** Clean:
- ~dfd clean~ deletes all refs under ~refs/dfd/snapshots/~
- Also runs auto-expiry (deletes refs older than 2 weeks) on any startup

** Expiration

Refs protect commits from GC, so we need explicit cleanup.

Decision: Time-based expiration, 2 weeks, async on startup.
- On startup, check commit dates of snapshot refs
- Delete refs pointing to commits older than 2 weeks
- Run async so it doesn't block the user
- Manual reset also available via ~dfd clean~

** Decisions

- ~--all~ mode: Not persisted. User must pass ~--all~ again if they want
  untracked files. If they don't, old untracked files appear as "deletions"
  from the last snapshot, which makes it obvious they need to re-run with ~--all~.
- Expiration: Time-based, 2 weeks, async cleanup on startup.
- Visual indicator: None needed - the presence of existing "Diff N" entries
  makes it clear you're continuing from a previous session.

* Snapshot Parent Chain (Refinement)

** Problem

Current implementation creates dangling commits with no parent. This means:
- Can't use ~git log~ to traverse snapshot history
- Need separate mechanism to fetch commit dates
- Inconsistent with how we display commits in log view

** Solution: Parent Chain

Snapshots should form a proper commit chain with parent relationships:

#+BEGIN_EXAMPLE
BASE_REF <- S0 <- S1 <- S2 <- ...
#+END_EXAMPLE

Where:
- S0's parent = BASE_REF (the commit we're initially diffing against)
- S1's parent = S0
- S2's parent = S1
- etc.

Benefits:
- ~git log <last-snapshot>~ naturally traverses the chain
- Commit dates come from git metadata (consistent with log view)
- Can use existing LogMetaOnly/LogPathsOnly infrastructure
- ~git diff S0^ S0~ shows the initial working tree changes (same as original diff)

** Determining the Base Ref

The base ref depends on how dfd was invoked:

| Invocation           | Base Ref | Notes                                    |
|----------------------+----------+------------------------------------------|
| ~dfd~                | ???      | Unstaged: working tree vs index          |
| ~dfd --cached~       | HEAD     | Index vs HEAD                            |
| ~dfd HEAD~           | HEAD     | Working tree vs HEAD                     |
| ~dfd <ref>~          | <ref>    | Working tree vs specific ref             |
| ~dfd <ref1> <ref2>~  | N/A      | Two fixed refs, snapshots don't apply    |

** Design Decisions

*** Default Behavior Change: Diff vs HEAD

*Decision*: Change default ~dfd~ behavior to diff working tree vs HEAD.

Current git behavior:
- ~git diff~ = working tree vs index (unstaged changes)
- ~git diff --cached~ = index vs HEAD (staged changes)
- ~git diff HEAD~ = working tree vs HEAD (all uncommitted)

New dfd behavior:
- ~dfd~ = working tree vs HEAD (enables snapshots by default)
- ~dfd --cached~ = index vs HEAD (no snapshots, index-focused)
- ~dfd <flag>~ = working tree vs index (no snapshots, for explicit unstaged view)

Rationale: Most users care about "what changed since last commit" more than
the staged/unstaged distinction. This also enables automatic snapshot
continuation by default.

*** Flag for "Unstaged Only" (Working Tree vs Index)

*Decision*: Use ~--unstaged~ flag for working tree vs index diff.

- ~dfd --unstaged~ = working tree vs index (git's default behavior)
- No snapshots available in this mode (index is not a commit)

*** Snapshot Control: --snapshots / --no-snapshots

*Decision*: Use ~--snapshots~ (default) and ~--no-snapshots~ flags.

- ~dfd~ = snapshots enabled (default), auto-continues if history exists
- ~dfd --no-snapshots~ = plain diff, ignore snapshot refs entirely
- ~dfd --snapshots~ = explicit enable (same as default, for clarity)

Removes the ~-c~ / ~--continue~ flag. Continuation is automatic.

*** Automatic Continuation

*Decision*: Automatically continue when snapshots exist for current base ref.

Implementation:
- Resolve the base ref to a SHA (e.g., HEAD -> abc123)
- Store snapshots keyed by base SHA: ~refs/dfd/snapshots/<base-sha>/0~
- On startup, check for existing snapshots for current base SHA
- If found, load history and take new snapshot as latest
- If not found, start fresh snapshot chain

This means:
- ~dfd~ against same HEAD automatically shows previous snapshot history
- ~dfd~ against different HEAD (after commit/checkout) starts fresh
- ~dfd clean~ clears all snapshot history for all bases
- ~dfd --no-snapshots~ shows plain diff without any snapshot behavior

*** Snapshots Not Available for Index-Based Diffs

*Decision*: ~--cached~ and unstaged-only modes don't support snapshots.

Rationale:
- Snapshots capture working tree state
- Need a commit as parent for snapshot chain
- Index is not a commit

** Open Questions

*** What happens when user commits?

*Decided*: Old snapshots become orphaned but not lost.

If user creates a real git commit while dfd is running:
- HEAD changes, so next ~dfd~ session sees different base SHA
- Old snapshot history remains under ~refs/dfd/snapshots/<old-sha>/~
- Running ~dfd~ after commit starts fresh (new base SHA)
- Auto-expiry (2 weeks) eventually cleans up old chains
- ~dfd clean~ removes all snapshots across all bases

*** Snapshot Labeling

*Decision*: All snapshots use consistent label format stored in commit message.

Format: "dfd: <base-sha> @ <datetime>"

Examples:
#+BEGIN_EXAMPLE
▶◯         +15  -3   2 Feb 5 09:15 dfd: d4e5f6a @ Feb 5 09:15
 ◯ a1b2c3d  +8  -1   1 Feb 5 09:10 dfd: d4e5f6a @ Feb 5 09:10
 ◯ e4f5g6h +22  -0   3 Feb 5 09:05 dfd: d4e5f6a @ Feb 5 09:05
#+END_EXAMPLE

- SHA column shows the snapshot commit SHA (unique identifier)
- Subject read from commit message (single source of truth)
- Newest at top (like log view)

*** Everything is a Snapshot

*Decision*: No separate "working tree" concept. Every view is a snapshot.

On startup:
1. Take snapshot immediately (captures current working tree)
2. This becomes the latest entry in the history
3. All entries are snapshots, displayed consistently

This simplifies the model: the view is always a log of snapshots, newest first.

*** Filtering with Path/Exclude Arguments

*Complication*: User may run dfd with different arguments against same base ref.

Examples:
- ~dfd -- src/file.go~ (only show file.go)
- ~dfd --exclude=vendor/~ (exclude vendor directory)
- ~dfd -- '*.go'~ (only Go files)

The snapshots capture the *full* working tree, but display should respect filters.

*Decision*: Apply filters to snapshot history, consistent with ~git log -- path~.

- Snapshots always capture full tree (for consistency across sessions)
- When displaying, use ~git log <last-snapshot> -- <paths>~ to get relevant snapshots
- Only snapshots that touched the filtered paths are shown
- Snapshots where the filtered file didn't change are omitted (same as git log)

Implementation:
- Pass path/exclude args through to git log when fetching snapshot history
- Since snapshots form a parent chain, git log naturally filters
- ~git log S3 -- file.go~ returns only S1, S3 if file.go only changed in those
- Each snapshot diff also filtered: ~git diff S0 S1 -- file.go~

This is simple because we leverage git's existing behavior.

*** Ref Storage Structure

*Decision*: Key snapshots by base SHA.

Current: ~refs/dfd/snapshots/0~, ~refs/dfd/snapshots/1~, etc.

New (keyed by base SHA):
#+BEGIN_EXAMPLE
refs/dfd/snapshots/<base-sha>/0 -> S0
refs/dfd/snapshots/<base-sha>/1 -> S1
refs/dfd/snapshots/<base-sha>/2 -> S2
#+END_EXAMPLE

Benefits:
- Multiple independent snapshot histories for different base refs
- Switching branches/commits doesn't lose other history
- ~dfd clean~ can clear all or be extended to clear specific base

*** Startup Behavior with Existing Snapshots

*Decision*: Take new snapshot on startup and show full history.

When running ~dfd~ with existing snapshots [S0, S1, S2]:
1. Load existing snapshot chain from refs
2. Take new snapshot S3 in background (parent = S2)
3. Display combined view:
   - "Diff 3" = S2 → S3 (changes since last session) - *expanded*
   - "Diff 2" = S1 → S2 - folded
   - "Diff 1" = S0 → S1 - folded
4. User presses R → take S4, show S3 → S4 at top

This combines diff + log behavior: showing incremental snapshot history
as a series of commits, with the most recent changes expanded.

*** Outstanding Refinements

**** DONE Use commit message for display label

- ~ListSnapshotRefs~ returns ~[]SnapshotInfo~ with SHA, Subject, Date
- Uses ~git log --format=%H%x00%s%x00%ci~ to get all info in one call
- No separate ~git show~ calls needed for each snapshot

- Consistent with how log view works

**** DONE Single ref instead of multiple refs per snapshot
Changed from refs/dfd/snapshots/<base-sha>/0, /1, /2 to single ref per base.

Implementation:
- ~refs/dfd/snapshots/<base-sha>~ points to latest snapshot
- ~git log --ancestry-path <base>..<ref>~ traverses parent chain
- ~UpdateSnapshotRef~ replaces ~CreateSnapshotRef~ (no index parameter)
- Much simpler: one ref to manage per base, delete/update is trivial

Note: Can't expire individual snapshots mid-chain (whole chain or nothing).
This is acceptable since expiry is time-based on the ref's commit date.
