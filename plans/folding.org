* Folding Feature

** Overview

Three-level file folding with per-file and all-files toggle controls.

** Fold Levels

Three levels that cycle:

1. *Normal* (default): Current hunk-based view with gaps between hunks
2. *Expanded*: Full file contents side-by-side, with diff parts highlighted
3. *Folded*: Only the header line visible (no trailing "=" characters, no leading newline)

Cycle order: Normal → Expanded → Folded → Normal

The starting level is configurable (default: Normal). Implementation should not hardcode
the default - use a constant/config that can be changed later.

** Keybindings

*** Tab (per-file)

- Affects the "current" file (determined by scroll position, shown in status bar)
- Cycles to the next fold level

*** Shift-Tab (all files)

- If files are NOT all on the same level → fold all to level 1 (Folded)
- If all files are on the same level → advance all to the next level (1→2→3→1)

** Visual Specifications

*** Folded Header

Same format as normal header but:
- No trailing "=" characters
- No leading newline (no blank line before)

Normal: ~═══ internal/tui/view.go ═══════════════════════~
Folded: ~═══ internal/tui/view.go~

*** Expanded View

- Both old and new file contents side-by-side
- All lines shown, including lines outside diff hunks
- Diff parts (added/removed) retain their colors (green/red)
- Context lines and non-diff lines use context styling
- Line numbers accurate to actual file positions
- No visual distinction needed between "gap" lines and context lines

*** Added/Deleted Files

In expanded mode:
- New file (no old version): only right side shown
- Deleted file (no new version): only left side shown

*** No Fold Indicators

No visual indicators (▸ ▾) for fold state - content difference speaks for itself.

** Search Integration

When fold state changes:
- Re-run the current search query automatically
- Update highlighted matches for the new visible content

** Performance Requirements

*** Async Loading

- File content fetching MUST be asynchronous
- User interaction must never be blocked
- No loading indicator needed (since it's non-blocking)

*** Concurrent Fetching

- Shift-Tab on many files should fetch all file contents concurrently
- Use goroutines with proper synchronization
- Cache results to avoid re-fetching on fold/unfold cycles

*** Existing Infrastructure

The ~pkg/content/Fetcher~ already supports:
- Lazy loading with caching
- All diff modes (show, diff, diff --cached, diff refs)
- Proper git commands for old/new versions

// internal/tui/commands.go (new file)

// FetchFileContent returns a command that fetches content for one file.
func (m Model) FetchFileContent(fileIndex int) tea.Cmd {
    return func() tea.Msg {
        fp := m.files[fileIndex]
        // Use m.fetcher to get content
        oldContent, _ := m.fetcher.GetOldContent(stripPrefix(fp.OldPath))
        newContent, _ := m.fetcher.GetNewContent(stripPrefix(fp.NewPath))
        return FileContentLoadedMsg{
            FileIndex:  fileIndex,
            OldContent: strings.Split(oldContent, "\n"),
            NewContent: strings.Split(newContent, "\n"),
        }
    }
}

// FetchAllFileContent returns a command that fetches content for all files concurrently.
func (m Model) FetchAllFileContent() tea.Cmd {
    return func() tea.Msg {
        var wg sync.WaitGroup
        results := make([]FileContent, len(m.files))

        for i := range m.files {
            wg.Add(1)
            go func(idx int) {
                defer wg.Done()
                fp := m.files[idx]
                old, _ := m.fetcher.GetOldContent(stripPrefix(fp.OldPath))
                new, _ := m.fetcher.GetNewContent(stripPrefix(fp.NewPath))
                results[idx] = FileContent{
                    FileIndex:  idx,
                    OldContent: strings.Split(old, "\n"),
                    NewContent: strings.Split(new, "\n"),
                }
            }(i)
        }

        wg.Wait()
        return AllContentLoadedMsg{Contents: results}
    }
}

**** 4.3 Per-file toggle logic

#+begin_src go
func (m Model) handleFoldToggle() (tea.Model, tea.Cmd) {
    // Find current file based on scroll position
    fileIdx := m.currentFileIndex()
    if fileIdx < 0 || fileIdx >= len(m.files) {
        return m, nil
    }

    fp := &m.files[fileIdx]
    newLevel := fp.FoldLevel.NextLevel()

    // If expanding and content not loaded, fetch it
    if newLevel == FoldExpanded && fp.OldContent == nil {
        fp.FoldLevel = newLevel
        m.calculateTotalLines()
        m.refreshSearch()
        return m, m.FetchFileContent(fileIdx)
    }

    fp.FoldLevel = newLevel
    m.calculateTotalLines()
    m.refreshSearch()
    return m, nil
}
#+end_src

**** 4.4 All-files toggle logic

#+begin_src go
func (m Model) handleFoldToggleAll() (tea.Model, tea.Cmd) {
    // Check if all files are on the same level
    if len(m.files) == 0 {
        return m, nil
    }

    firstLevel := m.files[0].FoldLevel
    allSame := true
    for _, fp := range m.files[1:] {
        if fp.FoldLevel != firstLevel {
            allSame = false
            break
        }
    }

    var newLevel FoldLevel
    if allSame {
        // All same - advance to next level
        newLevel = firstLevel.NextLevel()
    } else {
        // Not same - collapse all to Folded
        newLevel = FoldFolded
    }

    // Apply to all files
    for i := range m.files {
        m.files[i].FoldLevel = newLevel
    }

    m.calculateTotalLines()
    m.refreshSearch()

    // If expanding to full view, fetch all content concurrently
    if newLevel == FoldExpanded {
        return m, m.FetchAllFileContent()
    }

    return m, nil
}

func buildFullFilePairs(fp FilePair) []displayRow {
    old := fp.OldContent
    new := fp.NewContent

    // Handle deleted file (no new content)
    if len(new) == 0 && len(old) > 0 {
        var rows []displayRow
        for i, line := range old {
            rows = append(rows, displayRow{
                pair: LinePair{
                    Left:  Line{Num: i + 1, Content: line, Type: Context},
                    Right: Line{Num: 0, Content: "", Type: Empty},

        return rows

                },
            })

** Scroll Position Stability on Fold Changes

*** Problem

When folding or collapsing, the user can lose their position - ending up looking
at a different file than before.

*** Desired Behavior

- If you were on a line which is still visible in the next level, jump to that same line
- If you were on a line that's no longer visible, jump up to the closest chunk or file header that appears before that line
- If you were on a blank line between file headers, count as the file header below (except at the bottom, which counts as the last file)

*** Resolved Questions

1. *What counts as "the line you were on"?*
   ANSWER: Fixed cursor position at 20% from top of viewport. The cursor is always
   at this visual position; scroll position changes to move content under it.

2. *Blank lines between file headers*
   ANSWER: Yes, blank separator lines count as the file below them.

   ANSWER: Jump to the nearest visible thing above. If that's a hunk separator, use that.
