* Lazy Commit Loading for Log Mode

** Goal

Load commits incrementally to support browsing unlimited git history without
upfront blocking or excessive memory usage.

** Phase 1: Skeleton-First Loading with Deferred Diffs

*** Current Behavior

1. =git log -p -n1000= fetches all metadata AND all diffs upfront
2. All diffs parsed synchronously before TUI starts
3. Blocks for seconds on large repos with many commits

*** Target Behavior

1. =git log= (no =-p=) fetches just metadata for all commits (fast, ~100KB)
2. TUI starts immediately with skeleton commits (=FilesLoaded=false=, =Files=[]=)
3. First viewport's worth of commits (~10-20) fetched with full diffs async
4. When user unfolds a commit, fetch its diff on-demand if not already loaded

*** Data Flow

#+begin_src
Startup:
  git log --format=<meta> -n1000
       ↓
  []CommitSet with FilesLoaded=false, Files=[]
       ↓
  TUI starts (instant)
       ↓
  Async: fetch first N commits' diffs

User scrolls/unfolds:
  CommitSet.FilesLoaded == false?
       ↓ yes
  git show -p <sha>
       ↓
  Parse diff → populate Files
       ↓
  Set FilesLoaded=true, recalculate rows
#+end_src

**** 4. Add async commit diff loading to TUI

New message types in internal/tui/messages.go (or commands.go):

#+begin_src go
type CommitDiffLoadedMsg struct {
    CommitIndex int
    Files       []sidebyside.FilePair
    TruncatedFileCount int
    Err         error
}

func (m Model) FetchCommitDiff(commitIdx int) tea.Cmd {
    return func() tea.Msg {
        sha := m.commits[commitIdx].Info.SHA
        diffStr, err := m.git.ShowDiff(sha)
        if err != nil {
            return CommitDiffLoadedMsg{CommitIndex: commitIdx, Err: err}
        }

        d, err := diff.Parse(diffStr)
        if err != nil {
            return CommitDiffLoadedMsg{CommitIndex: commitIdx, Err: err}
        }

        files, truncated := sidebyside.TransformDiff(d)
        return CommitDiffLoadedMsg{
            CommitIndex:        commitIdx,
            Files:              files,
            TruncatedFileCount: truncated,
        }
    }
}
#+end_src

**** 7. Pre-fetch first viewport on startup

In =NewWithCommits()= or via a startup command:

#+begin_src go
func (m Model) prefetchInitialCommits() tea.Cmd {
    // Calculate how many commits fit in viewport
    // (estimate ~3-5 visible rows per folded commit header)
    visibleCommits := m.height / 4
    if visibleCommits > len(m.commits) {
        visibleCommits = len(m.commits)
    }

    var cmds []tea.Cmd
    for i := 0; i < visibleCommits && i < maxPrefetch; i++ {
        if !m.commits[i].FilesLoaded {
            cmds = append(cmds, m.FetchCommitDiff(i))
        }
    }
    return tea.Batch(cmds...)
}
#+end_src

** Phase 2: Memory Eviction (Future)

*** Goal

Prevent unbounded memory growth when scrolling through very long history.

*** Approach: Distance-Based Eviction

When a commit is far from the viewport (e.g., 50+ commits away):
1. Force-fold the commit
2. Clear its =Files= slice (=Files = nil=)
3. Set =FilesLoaded = false=
4. Keep =CommitInfo= (skeleton) intact

Re-expanding triggers a fresh fetch.

*** Triggers

- After handling scroll events, check distance from viewport
- Use a hysteresis zone to avoid thrashing (e.g., evict at 50, reload at 40)

*** Considerations

- User jumping back with =gg= would trigger re-fetches
- Could keep an LRU cache of N recently-viewed commits
- May want to batch eviction checks (not on every scroll)

*** Deferred

This phase is deferred until Phase 1 is working and we have real-world
usage data on memory patterns.
