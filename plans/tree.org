#+TITLE: Tree Hierarchy Visual Indicator
#+DATE: 2025-01-25

* Overview

Adding tree-style visual indicators (like the =tree= command) to show the
hierarchy of commits, files, and content. Goal is to make it easier to see
structure when paging through lots of file data, especially when scrolling
through many files across multiple commits.

* Problem Statement

When viewing diffs with many files or multiple commits:
- Easy to lose track of which commit you're in
- Hard to see file boundaries when scrolling through content
- No visual indication of hierarchy (commit > file > hunk > line)
- Folded items don't clearly show their relationship to siblings

* Current Implementation (Prototype)

** Tree Characters Used
| Char | Name                 | Usage                              |
|------+----------------------+------------------------------------|
| =─=  | Horizontal line      | Branch connector, root indicator   |
| =├=  | Branch               | Node with more siblings below      |
| =└=  | Last branch          | Final node at this level           |
| =│=  | Vertical line        | Continuation (more children below) |

** Current Hierarchy (2 levels)

├─── ◐ file 1
│    │ structural diff rows

│    │ hunk separators
├─── ◐ file 2
│    │ ...
└─── ◐ file 3 (last)

** Current Prefix Width
- Tree prefix is 7 characters wide for content rows
- Format: =[commit-level][4 spaces][file-level][space]=
- Examples:
  - =│    │ = (7 chars) - non-last file, expanded
  - =     │ = (7 chars) - last file, expanded
  - =│      = (7 chars) - non-last file, folded
  - =       = (7 chars) - last file, folded

* Generic Tree Representation

** Why Generalize?

Current problems:
1. Tree logic is copy-pasted across 8+ render functions
2. Each function has its own =if isLastFileInCommit= / =if isFileFolded= logic
3. Adding hunks as a tree level would require modifying every function
4. No single source of truth for tree structure
5. Hard to change tree appearance (colors, characters, width)

** Proposed Data Model

#+begin_src go
// TreeLevel represents one level in the tree hierarchy
type TreeLevel struct {
    IsLast   bool           // Is this the last child at its level?
    IsFolded bool           // Is this node folded (hiding children)?
    Style    lipgloss.Style // Color/style for tree chars at this level
    Depth    int            // 0=commit, 1=file, 2=hunk, 3=content
}

// TreePath represents the full path from root to a node
type TreePath struct {
    Ancestors []TreeLevel // Parent levels (for continuation lines)
    Current   *TreeLevel  // This node's level (for branch char), nil for content
}
#+end_src

** Generic Rendering Functions

#+begin_src go
// renderTreeContinuation renders the vertical continuation lines for ancestors
// Example output: "│    │    " for depth 2 with no folded ancestors
func renderTreeContinuation(ancestors []TreeLevel) string {
    var b strings.Builder
    for _, level := range ancestors {
        if level.IsLast || level.IsFolded {
            b.WriteString("     ") // 5 spaces - no continuation
        } else {
            b.WriteString(level.Style.Render("│"))
            b.WriteString("    ") // │ + 4 spaces = 5 chars
        }
    }
    return b.String()
}

// renderTreeBranch renders the branch character for a header node
// Example output: "├───" or "└───"
func renderTreeBranch(level TreeLevel) string {
    if level.IsLast {
        return level.Style.Render("└───")
    }
    return level.Style.Render("├───")
}

// renderTreePrefix renders the full prefix for any row
// For headers: continuation + branch
// For content: continuation + extra │ for parent
func renderTreePrefix(path TreePath, isHeader bool) string {
    continuation := renderTreeContinuation(path.Ancestors)
    if isHeader && path.Current != nil {
        return continuation + renderTreeBranch(*path.Current)
    }
    // Content row - add one more continuation level
    if len(path.Ancestors) > 0 {
        last := path.Ancestors[len(path.Ancestors)-1]
        if last.IsFolded {
            return continuation + "     "
        }
        return continuation + last.Style.Render("│") + " "
    }
    return continuation
}
#+end_src

** Width Calculation

#+begin_src go
const (
    TreeLevelWidth = 5  // "│    " or "     " per level
    TreeBranchWidth = 4 // "├───" or "└───"
)

func treeWidth(depth int, isHeader bool) int {
    if isHeader {
        return (depth * TreeLevelWidth) + TreeBranchWidth
    }
    return ((depth + 1) * TreeLevelWidth) + 1 // +1 for space after last │
}
#+end_src

Hunks become first-class tree nodes:

type HunkInfo struct {

Benefits of foldable hunks:
- Collapse uninteresting hunks while reviewing
- See overview of all changes in a file
- Jump between hunks easily
- Reduce visual noise

* Building the Tree

** In buildRows()

#+begin_src go
func (m Model) buildRows() []displayRow {
    var rows []displayRow

    for commitIdx, commit := range m.commits {
        isLastCommit := commitIdx == len(m.commits)-1
        commitStyle := lipgloss.NewStyle().Foreground(lipgloss.Color("3")) // yellow

        commitLevel := TreeLevel{
            IsLast:   isLastCommit,
            IsFolded: commit.FoldLevel == sidebyside.CommitFolded,
            Style:    commitStyle,
            Depth:    0,
        }

        // Commit header row
        rows = append(rows, displayRow{
            kind: RowKindCommitHeader,
            treePath: TreePath{
                Ancestors: nil,
                Current:   nil, // root has no branch
            },
            // ...
        })

        // Build file rows with tree path
        for fileIdx := range filesInCommit {
            isLastFile := fileIdx == lastFileIdx
            fileStyle := fileStatusStyle(files[fileIdx])

            fileLevel := TreeLevel{
                IsLast:   isLastFile,
                IsFolded: files[fileIdx].FoldLevel == FoldFolded,
                Style:    fileStyle,
                Depth:    1,
            }

            // File header
            rows = append(rows, displayRow{
                kind: RowKindHeader,
                treePath: TreePath{
                    Ancestors: []TreeLevel{commitLevel},
                    Current:   &fileLevel,
                },
                // ...
            })

            // Content rows
            for _, pair := range filePairs {
                rows = append(rows, displayRow{
                    kind: RowKindContent,
                    treePath: TreePath{
                        Ancestors: []TreeLevel{commitLevel, fileLevel},
                        Current:   nil, // content has no branch
                    },
                    // ...
                })
            }
        }
    }
    return rows
}

* Color Scheme

** Current (Prototype)
- Commit header: yellow (color 3)
- Commit info: yellow (color 3)
- File tree lines: file status color (green=added, red=deleted, blue=modified)

** Proposed Scheme
| Level       | Element          | Color                    | Rationale                |
|-------------+------------------+--------------------------+--------------------------|
| Commit      | Header, borders  | Yellow (3)               | Matches git log          |
| Commit info | Header, lines    | Yellow (3) dim           | Subordinate to commit    |
| File        | Header, branch   | Status color             | Visual diff indicator    |
| File        | Continuation │   | Status color dim         | Less prominent           |
| Hunk        | Header           | Cyan (6)                 | Distinct from file       |
| Hunk        | Continuation │   | Gray (8)                 | Minimal visual noise     |
| Content     | Tree lines       | Gray (8)                 | Focus on content         |

* Diff View (No Commits)

** Current State
When viewing plain diff (=git diff=, no commit metadata):
- =m.commits= is empty or has no metadata
- Files are shown directly

** Options

1. *No tree* - files shown without any tree prefix
   #+begin_example
   ◐ src/main.go
   +added line
   ◐ src/util.go
   #+end_example

2. *Single-level tree* - files as roots with children
   #+begin_example
   ─ ◐ src/main.go
     │ +added line
   ─ ◐ src/util.go
   #+end_example

3. *Files as siblings* - implicit root
   #+begin_example
   ├─── ◐ src/main.go
   │    │ +added line
   └─── ◐ src/util.go
        │ +added line
   #+end_example

** Recommendation
Option 2 (single-level tree) - consistent visual language, clear that content
belongs to file, no phantom parent node.

* Migration Path

** Phase 1: Add Generic Infrastructure
1. Define =TreeLevel= and =TreePath= types
2. Add =treePath= field to =displayRow=
3. Create =renderTreePrefix()= function
4. Keep existing =isLastFileInCommit=, =isFileFolded= working

** Phase 2: Migrate Existing Render Functions
For each render function:
1. Populate =treePath= in =buildRows()=
2. Update render function to use =renderTreePrefix(row.treePath, isHeader)=
3. Remove direct tree character generation
4. Test that output matches

Order of migration:
1. =renderStructuralDiffRow= (simplest)
2. =renderLinePair= (most common)
3. =renderHunkSeparator= family
4. =renderHeader=
5. =renderCommitInfoHeader=
6. Remaining row types

** Phase 3: Add Hunk Level
1. Add =HunkInfo= struct to track hunks within files
2. Parse hunk boundaries in diff processing
3. Create =RowKindHunkHeader= row type
4. Add hunk fold state
5. Update =buildRows()= to include hunk level in tree path
6. Create =renderHunkHeader()=

** Phase 4: Cleanup
1. Remove =isLastFileInCommit=, =isFileFolded= fields
2. Remove duplicate tree logic from render functions
3. Update all tests
4. Update golden files

* Performance Considerations

** Memory
- =TreePath= adds ~48+ bytes per row (slice header + 2-3 TreeLevel structs)
- For 10,000 row diff: ~500KB additional
- Acceptable for typical usage

** Rendering
- =renderTreePrefix()= called once per visible row
- String concatenation is fast
- No regex or complex parsing
- Profile if issues arise

* Open Questions

1. Should right side of diff also show tree? (Probably no - cluttered)

2. How to handle the border =│= at end of file headers? Currently conflicts
   visually with tree =│=. Options:
   - Remove header border
   - Use different character
   - Accept the visual similarity

3. Should tree lines connect directly to fold icons (=◐=, =◯=)?
   Current: =├─── ◐= (space between)
   Alternative: =├───◐= (connected)

4. What about very deep nesting? (Commit > file > hunk > inline comment?)
   - Cap at 3-4 levels?
   - Auto-collapse deep levels?

5. Should folding a parent auto-fold children, or preserve their state?
