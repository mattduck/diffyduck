#+TITLE: Tree Hierarchy Visual Indicator
#+DATE: 2025-01-25

* Overview

Adding tree-style visual indicators (like the =tree= command) to show the
hierarchy of commits, files, and content. Goal is to make it easier to see
structure when paging through lots of file data, especially when scrolling
through many files across multiple commits.

* Problem Statement

When viewing diffs with many files or multiple commits:
- Easy to lose track of which commit you're in
- Hard to see file boundaries when scrolling through content
- No visual indication of hierarchy (commit > file > hunk > line)
- Folded items don't clearly show their relationship to siblings

* Current Implementation (Prototype)

** Tree Characters Used
| Char | Name                 | Usage                              |
|------+----------------------+------------------------------------|
| =─=  | Horizontal line      | Branch connector, root indicator   |
| =├=  | Branch               | Node with more siblings below      |
| =└=  | Last branch          | Final node at this level           |
| =│=  | Vertical line        | Continuation (more children below) |

- Tree prefix is 7 characters wide for content rows
- Format: =[commit-level][4 spaces][file-level][space]=
- Examples:
  - =│    │ = (7 chars) - non-last file, expanded
  - =     │ = (7 chars) - last file, expanded
  - =│      = (7 chars) - non-last file, folded
  - =       = (7 chars) - last file, folded

* Generic Tree Representation

** Why Generalize?

Current problems:
1. Tree logic is copy-pasted across 8+ render functions
2. Each function has its own =if isLastFileInCommit= / =if isFileFolded= logic
3. Adding hunks as a tree level would require modifying every function
4. No single source of truth for tree structure
5. Hard to change tree appearance (colors, characters, width)

** Proposed Data Model

#+begin_src go
// TreeLevel represents one level in the tree hierarchy
type TreeLevel struct {
    IsLast   bool           // Is this the last child at its level?
    IsFolded bool           // Is this node folded (hiding children)?
    Style    lipgloss.Style // Color/style for tree chars at this level
    Depth    int            // 0=commit, 1=file, 2=hunk, 3=content
}

// TreePath represents the full path from root to a node
type TreePath struct {
    Ancestors []TreeLevel // Parent levels (for continuation lines)
    Current   *TreeLevel  // This node's level (for branch char), nil for content
}

** Generic Rendering Functions

#+begin_src go
// renderTreeContinuation renders the vertical continuation lines for ancestors
// Example output: "│    │    " for depth 2 with no folded ancestors
func renderTreeContinuation(ancestors []TreeLevel) string {
    var b strings.Builder
    for _, level := range ancestors {
        if level.IsLast || level.IsFolded {
            b.WriteString("     ") // 5 spaces - no continuation
        } else {
            b.WriteString(level.Style.Render("│"))
            b.WriteString("    ") // │ + 4 spaces = 5 chars
        }
    }
    return b.String()
}

// renderTreeBranch renders the branch character for a header node
// Example output: "├───" or "└───"
func renderTreeBranch(level TreeLevel) string {
    if level.IsLast {
        return level.Style.Render("└───")
    }
    return level.Style.Render("├───")
}

// renderTreePrefix renders the full prefix for any row
// For headers: continuation + branch
// For content: continuation + extra │ for parent
func renderTreePrefix(path TreePath, isHeader bool) string {
    continuation := renderTreeContinuation(path.Ancestors)
    if isHeader && path.Current != nil {
        return continuation + renderTreeBranch(*path.Current)
    }
    // Content row - add one more continuation level
    if len(path.Ancestors) > 0 {
        last := path.Ancestors[len(path.Ancestors)-1]
        if last.IsFolded {
            return continuation + "     "
        }
        return continuation + last.Style.Render("│") + " "
    }
    return continuation
}
#+end_src

** Width Calculation

#+begin_src go
const (
    TreeLevelWidth = 5  // "│    " or "     " per level
    TreeBranchWidth = 4 // "├───" or "└───"
)

func treeWidth(depth int, isHeader bool) int {
    if isHeader {
        return (depth * TreeLevelWidth) + TreeBranchWidth
    }
    return ((depth + 1) * TreeLevelWidth) + 1 // +1 for space after last │
}

func (m Model) buildRows() []displayRow {
    var rows []displayRow

    for commitIdx, commit := range m.commits {
        isLastCommit := commitIdx == len(m.commits)-1
        commitStyle := lipgloss.NewStyle().Foreground(lipgloss.Color("3")) // yellow

        commitLevel := TreeLevel{
            IsLast:   isLastCommit,
            IsFolded: commit.FoldLevel == sidebyside.CommitFolded,
            Style:    commitStyle,
            Depth:    0,
        }

        // Commit header row
        rows = append(rows, displayRow{
            kind: RowKindCommitHeader,
            treePath: TreePath{
                Ancestors: nil,
                Current:   nil, // root has no branch
            },
            // ...
        })

        // Build file rows with tree path
        for fileIdx := range filesInCommit {
            isLastFile := fileIdx == lastFileIdx
            fileStyle := fileStatusStyle(files[fileIdx])

            fileLevel := TreeLevel{
                IsLast:   isLastFile,
                IsFolded: files[fileIdx].FoldLevel == FoldFolded,
                Style:    fileStyle,
                Depth:    1,
            }

            // File header
            rows = append(rows, displayRow{
                kind: RowKindHeader,
                treePath: TreePath{
                    Ancestors: []TreeLevel{commitLevel},
                    Current:   &fileLevel,
                },
                // ...
            })

            // Content rows
            for _, pair := range filePairs {
                rows = append(rows, displayRow{
                    kind: RowKindContent,
                    treePath: TreePath{
                        Ancestors: []TreeLevel{commitLevel, fileLevel},
                        Current:   nil, // content has no branch
                    },
                    // ...
                })
            }
        }
    }
    return rows
}

* Unified Node Architecture (Analysis)

** Problem Statement

The current implementation has three representations for file headers:
1. *Single-line* (folded): Just the header text with fold icon
2. *Two-line* (unfolded without border): Header + content
3. *Three-line* (unfolded with border): Top border + header + bottom border

Similarly, commit headers and commit-info nodes each have their own rendering logic.
This leads to duplicated concepts scattered across render functions.

** Duplicated Concepts

These concepts are repeated for each node type (commit, commit-info, file, hunk):

| Concept              | Current Implementation                          | Notes                          |
|----------------------+-------------------------------------------------+--------------------------------|
| Fold state           | =FoldLevel=, =CommitFoldLevel= (separate enums) | Could be unified enum          |
| Border rendering     | =renderHeaderTopBorder=, =renderCommitHeader*=  | Similar logic, different style |
| Tree path building   | Inline in =buildRows()= for each type           | Copy-paste patterns            |
| Cursor rendering     | Arrow position varies by node type              | Could be generic               |
| Margin handling      | Blank rows added inconsistently                 | Needs unified approach         |
| Header mode          | Implicit in =borderVisible= + =FoldLevel=       | Should be explicit             |

** Node-Specific Concepts

These remain legitimately different per node type:

| Node Type   | Specific Data                                        |
|-------------+------------------------------------------------------|
| Commit      | Hash, author, date, subject, tabular grid rendering  |
| Commit-info | Metadata fields (author, date, message body)         |
| File        | Path, status (+/-/~), fold level, content, stats     |
| Hunk        | Header line (@@), line ranges, content slice         |

// determineHeaderMode computes mode based on fold state and prev sibling
func determineHeaderMode(nodeType NodeType, isFolded bool, isFirstChild bool, prevSiblingUnfolded bool) HeaderMode {
    if isFolded {
        return HeaderSingleLine
    }

    // Unfolded
    if isFirstChild || prevSiblingUnfolded {
        return HeaderThreeLine
    }

    // Prev sibling is folded
    if nodeType == NodeCommit {
        return HeaderSingleLine // Commits fall back to single-line
    }
    return HeaderTwoLine // Files/commit-info get two-line
}

** Decisions (Finalized)

1. *TreeNode*: Use struct, not interface. Can extract interface later if needed.

2. *Rows vs nodes*: Keep separate. Copy =treePath= and =headerMode= into =displayRow=
   at build time. Rows are view-layer cache; nodes are domain objects.

3. *Fold state*: Stays on the data (=file.FoldLevel=, =commit.FoldLevel=).
   Separate state map deferred until we need undo/redo.

*** Rules for Files and Commit-info

| Own State | Prev Sibling   | Mode        | Notes                              |
|-----------+----------------+-------------+------------------------------------|
| Folded    | (any)          | SingleLine  | Optional preview below, no margin  |
| Unfolded  | (first child)  | ThreeLine   | Draws top into parent's margin     |
| Unfolded  | Folded         | TwoLine     | No margin above, skip top border   |
| Unfolded  | Unfolded       | ThreeLine   | Draws top into prev's margin       |

Unfolded files/commit-info produce *1 margin line* below.

*** Rules for Commits

| Own State | Prev Sibling   | Mode        | Notes                              |
|-----------+----------------+-------------+------------------------------------|
| Folded    | (any)          | SingleLine  | No margin below                    |
| Unfolded  | (first commit) | ThreeLine   | Draws top into page margin         |
| Unfolded  | Folded         | SingleLine  | Fallback (no 2-line for commits)   |
| Unfolded  | Unfolded       | ThreeLine   | Draws top into prev's margin       |

Unfolded commits produce *2 margin lines* below:
- Line 1: Available for commit's own bottom border
- Line 2: Available for first child's top border

*** Preview Content

Preview (structural diff shown under folded files) does NOT count as unfolded.
A folded file with preview still produces no margin, and next sibling treats
it as "prev folded".

*** Key Invariant

The top border is only rendered when there's a margin line above to draw into.
This prevents layout shift when toggling fold state - content below doesn't move.

*** Content Shift Prevention

Unfolded items must always produce their full margin allocation, even when the
next sibling is folded. This means:

- Unfolded files: 1 margin line + 1 top border slot (renders blank if next is folded)
- Unfolded commits: 2 margin lines (for own bottom border + first child's top border slot)

The "slot" row is always present to maintain consistent row counts. It renders as
a border when the next item needs it (HeaderThreeLine), or as blank otherwise.
