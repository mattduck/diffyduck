* Treesitter Syntax Highlighting

** Overview

Add syntax highlighting using tree-sitter to make diffs easier to read.
Syntax colors should be the "base layer" that other highlights (search, cursor,
word-diff) modify on top of.

** Architecture

*** Layer 1: Semantic Categories

A universal set of semantic categories that all languages map to:

#+begin_src go
// pkg/highlight/categories.go

type Category int

const (
    CategoryNone Category = iota

    // Basics
    CategoryKeyword      // func, if, else, return, import, etc.
    CategoryOperator     // +, -, *, /, =, ==, etc.
    CategoryPunctuation  // (), {}, [], ;, etc.

    // Literals
    CategoryString       // "hello", 'c', `raw`
    CategoryNumber       // 42, 3.14, 0xFF
    CategoryBoolean      // true, false
    CategoryNil          // nil, null, None

    // Names
    CategoryFunction     // function/method names at definition
    CategoryFunctionCall // function/method names at call site
    CategoryType         // type names
    CategoryVariable     // variable names
    CategoryConstant     // constant names (ALL_CAPS convention)
    CategoryParameter    // function parameters
    CategoryField        // struct/object fields

    // Comments
    CategoryComment      // // or /* */
    CategoryDocComment   // /// or /** */ (if distinguishable)

    // Special
    CategoryTag          // struct tags, annotations
    CategoryAttribute    // @decorators, #[attributes]
    CategoryNamespace    // package names, module names
)
#+end_src

*** Layer 4: Highlighter

The main interface for getting highlight spans from source code.

#+begin_src go
// pkg/highlight/highlighter.go

// Span represents a highlighted range in source code.
type Span struct {
    Start    int      // byte offset
    End      int      // byte offset
    Category Category
}

// Highlighter provides syntax highlighting for source code.
type Highlighter struct {
    theme     Theme
    languages map[string]*LanguageConfig  // by extension
    parsers   map[string]*sitter.Parser   // cached parsers
    cache     *SpanCache                  // cached results
}

// Highlight returns spans for the given source code.
// Results are cached by content hash.
func (h *Highlighter) Highlight(filename string, content []byte) ([]Span, error)

// HighlightRange returns spans for a specific line range.
// Parses with context to ensure correct highlighting at boundaries.
func (h *Highlighter) HighlightRange(filename string, content []byte, startLine, endLine int) ([]Span, error)

** Performance Strategy

*** Non-Blocking Parsing

Parsing must not block user interaction. Use async pattern like content fetching:

#+begin_src go
// internal/tui/messages.go

type HighlightReadyMsg struct {
    FileIndex int
    Spans     []highlight.Span
}

// internal/tui/commands.go

func (m Model) RequestHighlight(fileIndex int) tea.Cmd {
    return func() tea.Msg {
        content := m.files[fileIndex].NewContent  // or OldContent
        filename := m.files[fileIndex].NewPath
        spans, _ := m.highlighter.Highlight(filename, []byte(strings.Join(content, "\n")))
        return HighlightReadyMsg{FileIndex: fileIndex, Spans: spans}
    }
}
#+end_src

*** Parse Visible + Context

Don't parse entire files upfront. Parse visible lines + buffer:

#+begin_example
Total file: 1000 lines
Visible: lines 100-150
Parse range: lines 80-170 (visible + 20 line buffer each side)

When scrolling to lines 200-250:
- If new range overlaps existing: extend parse
- If new range disjoint: parse new range, keep old cached
#+end_example

*** Caching Strategy

Multiple cache levels:

1. *Parse tree cache*: Tree-sitter parse trees by content hash
   - Trees can be reused if content unchanged
   - Tree-sitter supports incremental parsing for edits

2. *Span cache*: Highlight spans by (content hash, line range)
   - Avoid re-walking tree for same content
   - Invalidate on theme change

3. *Rendered line cache*: Final styled strings
   - Most expensive to compute (applies all style layers)
   - Invalidate on scroll, search, fold change

** Style Composition

*** Layer Order (bottom to top)

1. *Syntax highlighting* (base layer)
   - Colors from treesitter based on node types

2. *Diff line type* (add/remove/context)
   - May adjust foreground slightly for removed lines
   - Could desaturate syntax colors on removed lines

3. *Word diff*
   - Inverts or highlights changed portions within a line
   - Overrides syntax colors in changed spans

4. *Search matches*
   - Yellow background for matches
   - Brighter yellow for current match
   - Completely overrides underlying colors

5. *Cursor highlight*
   - Only affects gutter areas, not content
   - Doesn't interact with syntax highlighting

*** Implementation

#+begin_src go
// Composing styles for a line:

func (m Model) styleLine(line string, lineNum int, syntaxSpans []Span) string {
    // Start with syntax highlighting
    styled := applySyntaxSpans(line, syntaxSpans, m.theme)

    // Apply word diff if this line has inline changes
    if inlineSpans := m.getInlineDiffSpans(lineNum); len(inlineSpans) > 0 {
        styled = applyInlineSpans(styled, inlineSpans)
    }

    // Apply search highlights if any matches on this line
    if searchSpans := m.getSearchSpans(lineNum); len(searchSpans) > 0 {
        styled = applySearchSpans(styled, searchSpans, m.currentMatch)
    }

    return styled
}
#+end_src

** Resolved Questions

1. *Incremental parsing*: No - only relevant for edits, we're read-only.

2. *Theme customization*: Hardcode for now, keep colors in one place for easy
   tweaking. Add user customization later if needed.

3. *Language detection*: File extension/suffix only to start.

4. *Diff-specific styling*: Keep same syntax colors on both sides. Diff info
   (add/remove indicators) will move to the gutter only (separate task before
   treesitter work). This simplifies composition since syntax highlighting
   applies uniformly to content.
