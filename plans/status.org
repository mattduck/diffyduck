* Status Command

Print-and-exit subcommand (`dfd status`, alias `s`) showing a rich overview
of the working tree: branch context, staged/unstaged file changes with
per-file structural diffs, and untracked files.

** Design

*** Package layout

New package: ~pkg/status/~
- ~render.go~ — build the full status output string
- ~render_test.go~ — unit tests for rendering

Alternatively, the rendering could live in ~cmd/dfd/~ like ~runBranches~
does for branches, but since there's meaningful logic (structural diff
per-file, section formatting), a dedicated package keeps it testable.

*** Data flow

#+begin_example
1. git.LocalBranches()         → branches.BuildTree() → branches.Render()
2. git.Diff("--cached")        → diff.Parse()         → staged FilePairs
3. git.Diff()                  → diff.Parse()         → unstaged FilePairs
4. git.ListUntrackedFiles()    → untracked paths
5. For each staged/unstaged file:
   a. Fetch old/new content     → git.GetFileContent(ref, path)
   b. Extract structure         → structure.Extract(content, lang)
   c. Compute structural diff   → structure.ComputeDiff(old, new, added, removed)
   d. Sort + truncate to 5      → render symbols
6. Assemble sections → print
#+end_example

*** Git operations needed

All existing:
- ~g.Diff("--cached")~ — staged diff (already supported via variadic args)
- ~g.Diff()~ — unstaged diff (already supported)
- ~g.ListUntrackedFiles()~ — untracked files (exists)
- ~g.GetFileContent(ref, path)~ — file content for structure extraction
- ~g.LocalBranches()~ + ~branches.BuildTree/Render~ — branch tree

For content fetching to compute structural diffs:
- Staged: old = ~HEAD:path~, new = ~:path~ (index)
- Unstaged: old = ~:path~ (index), new = working tree (read file directly)
- If nothing staged, unstaged old = ~HEAD:path~

We may want a helper that determines old/new refs based on whether a file
is staged, unstaged, or both.

*** Reusable code

From ~pkg/structure/~:
- ~Extract()~ — parse file content to get structure map
- ~ComputeDiff()~ — compute structural diff
- ~ElementChange~, ~ChangeKind~ — types

From ~internal/tui/view_commit.go~ (logic to extract, not TUI rendering):
- Tree-building logic (group methods under parent types)
- Sort by total lines changed (descending)
- Truncation with ~...(N more)~

We'll port the tree-building + sorting + truncation logic into a shared
helper (probably in ~pkg/structure/~) so both TUI and status command can
use it without duplication.

From ~pkg/diff/~:
- ~Parse()~ — parse unified diff
- ~File~, ~Hunk~, ~Line~ — types for extracting added/removed line numbers

From ~pkg/branches/~:
- ~BuildTree()~, ~Render()~ — direct reuse

*** Rendering approach

ANSI terminal output (like branches), not Bubble Tea. Use direct ANSI
escape codes or lipgloss for color. Match the diff view's color conventions:
- Green for added (~+~)
- Red for removed (~-~)
- Blue for modified (~~)
- Cyan for renamed (~→~)
- Dim/dark green+red for structural diff line counts
- fg=8 for keyword in structural diff (~func~, ~type~)
- Section headers in bold or bright white

*** Step 2: Add ~pkg/status/~ package

~pkg/status/render.go~:
- ~Render(branches string, staged, unstaged []FileStatus, untracked []string) string~
- ~FileStatus~ struct: Path, OldPath, Status, Added, Removed, Symbols []SymbolLine
- ~SymbolLine~ struct: Kind, Signature, Added, Removed, IsTruncation, TruncCount
- Section rendering helpers (colorized file lines, symbol lines)

*** Step 3: Wire up in ~cmd/dfd/main.go~

- Add ~status~ subcommand + ~s~ alias in argument parsing
- Add ~usageStatus~ const for help text
- Add ~runStatus()~ function:
  1. Get branch tree (reuse ~runBranches~ logic)
  2. Get staged diff (~g.Diff("--cached")~), parse, compute structure
  3. Get unstaged diff (~g.Diff()~), parse, compute structure
  4. Get untracked files
  5. Render and print

*** Step 4: Content fetching for structural diffs

For each file in staged/unstaged diffs, fetch old and new content to
extract structure:

Staged files:
- Old: ~g.GetFileContent("HEAD", path)~
- New: ~g.GetFileContent("", path)~ (index/staged version)

Unstaged files:
- Old: ~g.GetFileContent("", path)~ (index version)
- New: read from working tree (~os.ReadFile~)

Handle edge cases:
- New files (old = empty)
- Deleted files (new = empty)
- Binary files (skip structure)
- Files that exist in both staged and unstaged (show in both sections)

** Open questions / decisions

- Should we detect files that appear in BOTH staged and unstaged and show
  them in both sections? (Yes — git status does this, e.g. partially staged
  files.) The structural diff in each section reflects only that section's
  changes.
- Performance: structural diff requires fetching file content and running
  treesitter. For a status command this should be fast enough since we're
  not rendering a TUI, but we could parallelise file processing if needed.
- Should ~dfd status~ accept flags? For v1, no flags. Later: ~--no-structure~
  to skip treesitter, ~--short~ for compact output.
